# [Plan] tamper-field-detection

> **Feature**: 위변조 편집 위치·유형 분류로 가짜 영수증 vs 개인정보 보호 처리 구분
> **Date**: 2026-04-22
> **Phase**: Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 현재 위변조 탐지는 "편집 유무"만 판단 — 카드번호 가림(정상)과 거래일시 조작(가짜)을 구분하지 못함 |
| **Solution** | Gemini에 편집 위치·유형 분류 추가(주 신호) + 픽셀 시그니처 분석으로 교체/은닉 구분(보조 신호) |
| **Functional UX Effect** | 카드번호·소지자명 가림 영수증은 승인, 거래일시·금액 조작 영수증은 반려 — 오탐 감소 |
| **Core Value** | 개인정보 보호 처리를 존중하면서 실제 위변조만 정밀 탐지 |

---

## Context Anchor

| 항목 | 내용 |
|------|------|
| **WHY** | 그림판으로 거래일시를 교체한 영수증이 승인되는 오류 발생 — 편집 위치·유형 구분 없이 면적 비율만 체크하는 한계 |
| **WHO** | 여신티켓 어드민 운영자 |
| **RISK** | Gemini가 편집 필드를 잘못 분류하면 가짜 영수증이 통과할 수 있음 → 픽셀 시그니처 보조 신호로 이중 검증 |
| **SUCCESS** | 거래일시 교체 편집 → 반려 / 카드번호 모자이크 처리 → 승인 |
| **SCOPE** | `background.js` TAMPER_PROMPT + parseTamperResult, `content.js` analyzeImageNoise + step3Tamper |

## 1. 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-01 | Gemini가 `편집부위`(없음\|카드정보\|거래핵심정보)와 `편집유형`(없음\|은닉\|교체)을 출력 | Must |
| FR-02 | `편집부위: 거래핵심정보` → 편집유형 무관 반려 (거래일시·금액·가맹점·승인번호·사업자번호) | Must |
| FR-03 | `편집부위: 카드정보` + `편집유형: 은닉` → 개인정보 보호 처리 → 통과 | Must |
| FR-04 | `편집부위: 카드정보` + `편집유형: 교체` → 카드번호 조작 의심 → 반려 | Must |
| FR-05 | 픽셀 분석에서 흰색 균일 영역(밝기 > 220, 분산 ≈ 0) 탐지 시 `교체` 힌트 반환 | Should |
| FR-06 | Gemini 필드 분류 불확실 시 픽셀 시그니처 보조 신호 활용 | Should |
| FR-07 | 추가 Gemini 호출 없음 — 기존 TAMPER_PROMPT에 필드 추가 방식 | Must |

---

## 2. 판정 로직

```
편집부위: 거래핵심정보          → [3단계] 반려 (유형 무관)
편집부위: 카드정보
  └─ 편집유형: 은닉            → 개인정보 보호 처리, 통과
  └─ 편집유형: 교체            → [3단계] 반려 (카드번호 조작 의심)
편집부위: 없음                  → 기존 tamperLevel 판정 유지

※ Gemini 편집부위 분류 실패(없음) + 픽셀 시그니처 '교체' 감지
  → 기존 tamperLevel medium/high 조건과 AND 처리
```

## 3. 필드 정의

| 분류 | 포함 항목 |
|------|----------|
| **거래핵심정보** | 거래일시, 결제금액(공급가·부가세·합계), 가맹점명, 사업자번호, 승인번호 |
| **카드정보** | 카드번호(마스킹 포함), 소지자명, 유효기간, 카드사명 |

## 4. 구현 범위

### 변경 파일
- `background.js` — TAMPER_PROMPT 수정 (2개 필드 추가 + 판별 기준)
- `background.js` — `parseTamperResult()` 신규 필드 파싱
- `content.js` — `analyzeImageNoise()` 흰색 교체 시그니처 탐지 추가
- `content.js` — `step3Tamper()` 판정 로직 교체

## 5. 성공 기준

- [ ] 거래일시 흰색 덮어쓰기 편집 영수증 → `[3단계]` 반려
- [ ] 카드번호 블러·모자이크·검정 처리 영수증 → 통과
- [ ] 카드번호 + 거래일시 동시 편집 → `[3단계]` 반려 (핵심 정보 편집 우선)
- [ ] 편집 없는 정상 영수증 → 기존 판정 로직 유지
- [ ] 추가 Gemini 호출 없음
