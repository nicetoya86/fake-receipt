# [Plan] receipt-image-validation

> **Feature**: 비영수증 이미지 조기 감지 및 반려
> **Date**: 2026-04-22
> **Phase**: Plan

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 광고 배너·사진·스크린샷 등 영수증이 아닌 이미지가 업로드되면 OCR·위변조 분석까지 모두 실행된 뒤 뒤늦게 반려되거나, 사유 없이 오탐 반려됨 |
| **Solution** | OCR Gemini 호출에 `영수증_여부` 판별 필드를 추가해 별도 API 비용 없이 OCR 직후 즉시 반려 |
| **Functional UX Effect** | 비영수증 이미지 업로드 시 명확한 안내 메시지와 함께 즉시 반려 → 운영자 혼란 감소 |
| **Core Value** | 검증 파이프라인 진입 조건 강화로 오탐·불필요한 API 비용 동시 절감 |

---

## Context Anchor

| 항목 | 내용 |
|------|------|
| **WHY** | 비영수증 이미지에서 픽셀 노이즈 오탐 등 무관한 사유로 반려되는 문제 발생 — 조기에 차단해야 함 |
| **WHO** | 여신티켓 어드민 운영자 — 영수증만 업로드해야 하는 실무자 |
| **RISK** | 영수증 인식 기준이 너무 엄격하면 구겨지거나 흐릿한 실제 영수증이 오탐 반려될 수 있음 |
| **SUCCESS** | 비영수증 이미지 업로드 시 `[0단계]` 메시지로 즉시 반려, 정상 영수증은 통과 |
| **SCOPE** | `background.js` OCR 프롬프트 수정 + `content.js` verifyReceipt 흐름에 판별 로직 추가 |

## 1. 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-01 | OCR Gemini 응답에 `영수증_여부: 예\|아니오` 필드 추가 — 추가 API 호출 없음 | Must |
| FR-02 | `영수증_여부: 아니오`이면 OCR 직후 즉시 반려, 이후 모든 단계 스킵 | Must |
| FR-03 | 반려 메시지: `[0단계] 영수증 이미지가 아닙니다 — 카드 영수증 이미지를 업로드해 주세요.` | Must |
| FR-04 | 영수증으로 인정하는 범위: POS 영수증, 신용·체크카드 전표, 세금계산서, 배달 영수증 | Must |
| FR-05 | 영수증이 아닌 것: 광고물, 사진, 스크린샷, 문서, 명함, 영수증과 무관한 인쇄물 | Must |
| FR-06 | OCR 재시도(STANDARD 2차, CAREFUL) 로직은 기존 유지 — 영수증 판별 결과만 1차에서 즉시 추출 | Must |

## 2. 검증 흐름 변경

### 기존 흐름
```
OCR → step1 → step2 → step3 → step4
```

### 변경 후 흐름
```
OCR (+ 영수증_여부 판별)
  ├─ 영수증_여부: 아니오 → 🔴 [0단계] 즉시 반려
  └─ 영수증_여부: 예   → step1 → step2 → step3 → step4
```

---

## 3. OCR 프롬프트 변경

### STANDARD_PROMPT 응답 형식 변경

**기존 (3줄)**:
```
사업자번호: XXX-XX-XXXXX
승인번호: XXXXXXXX
카드BIN: XXXXXX
```

**변경 후 (4줄)**:
```
영수증여부: 예|아니오
사업자번호: XXX-XX-XXXXX
승인번호: XXXXXXXX
카드BIN: XXXXXX
```

### 영수증 판별 기준 프롬프트 추가
- 결제 금액, 가맹점명, 날짜, 카드번호 중 2개 이상 포함 시 영수증으로 판단
- 위 요소가 전혀 없는 이미지(광고, 사진, 일반 문서 등)는 "아니오"

---

## 4. 구현 범위

### 변경 파일
- `background.js`
  - `STANDARD_PROMPT` — 응답 형식에 `영수증여부:` 줄 추가 + 판별 기준 명시
  - `geminiOCRFromDataURL()` — 1차 응답에서 `isReceipt` 추출 후 즉시 반환
  - `parseOCRResult()` (또는 인라인) — `isReceipt: boolean` 필드를 응답 객체에 추가
- `content.js`
  - `verifyReceipt()` — OCR 직후 `ocrRaw?.isReceipt === false` 체크 → 즉시 반려

### 변경 없는 파일
- `manifest.json`, `styles.css`, `options.js`
- BIN 데이터 파일

---

## 5. 성공 기준

- [ ] 광고 배너 이미지 → `[0단계]` 반려 메시지, step1~4 미실행
- [ ] 일반 사진(인물, 풍경) → `[0단계]` 반려 메시지
- [ ] 채팅 스크린샷 → `[0단계]` 반려 메시지
- [ ] 정상 영수증 (POS·카드전표) → 통과, step1 진행
- [ ] 구겨지거나 흐릿한 영수증 → 통과 (오탐 없음)
- [ ] 추가 Gemini API 호출 없음 (기존 OCR 1회 재사용)
